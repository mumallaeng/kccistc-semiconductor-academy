- Git과 GitHub를 끝까지 구분해서 말하기

- 권장 흐름
  - 문제 인식
  - Git 구조 이해
  - `.git` 설명
  - 로컬 사용

## 1. Version Control System 개요

- 형상 관리 시스템 정의
  - 코드 + 변경 이력 관리
  - 협업
  - 추적
  - 복구

- 왜 필요한가
  - 이전 버전 확인
  - 실수 복구
  - 여러 사람이 동시에 작업
  - 기능 단위 분리

- 발전 과정
  - Local VCS
    - 로컬에서만 관리
    - 협업 불가
    - 디스크 손실 시 전체 손실
  - Centralized VCS
    - 중앙 서버 기반 협업
    - 서버 의존성 높음
    - 오프라인 작업 불가
    - 중앙 서버 반영 시 바로 영향 가능
  - Distributed VCS
    - 로컬 저장소 포함
    - commit = 개인 저장
    - push = 공유 시점 제어
    - 오프라인 작업 가능

- 대표 도구
  - CVS
    - 초기 시스템
  - SVN
    - 중앙 집중형
    - 아직 일부 사용
  - Git
    - 분산형
    - 빠른 속도
    - 현재 표준

- 멘트
  - "Git이 강한 이유는 단순 저장이 아니라, 이력 관리와 분산 구조에 있습니다."

## 2. Git 핵심 구조 (3 영역)

- Git = 상태 스냅샷 기반 관리
  - 파일 하나하나보다
  - "이 시점의 프로젝트 상태"를 저장

- 영역 구성
  - Working Directory
    - 실제 작업 공간
    - 파일 수정 발생
  - Staging Area
    - commit 대상 준비 영역
    - `git add` 결과 저장
  - Repository
    - commit 저장소
    - 스냅샷 기록

- 흐름
  - 수정
  - add
  - commit

- 핵심 멘트
  - "Git은 파일을 복사해두는 느낌이 아니라, 상태를 기록하는 도구입니다."
  - "바로 commit 되는 게 아니라 staging을 거친다는 점이 중요합니다."

## 2-1. `.git` 폴더

- 정의
  - Git 저장소의 실체
  - 프로젝트의 히스토리 / 설정 / 참조 정보 저장

- 포함되는 것
  - commit 정보
  - branch 정보
  - HEAD 정보
  - remote 설정
  - 기타 Git 메타데이터

- 핵심 이해
  - Working Directory = 사람이 보는 파일
  - `.git` = Git이 보는 기록 저장소

- 연결 포인트
  - `git init` 하면 `.git` 생성
  - `git clone` 해도 `.git` 포함

- 복구 관점
  - 프로젝트 파일만 있고 `.git` 없으면 그냥 일반 폴더
  - `.git`이 살아 있으면 Git 히스토리 복구 가능

- 멘트
  - "Git을 쓴다는 말은 결국 이 숨은 `.git` 폴더를 같이 관리한다는 뜻입니다."
  - "나중에 복구 개념도 결국 여기로 다시 연결됩니다."

## 3. 기본 명령 (로컬 개인 작업)

- 시작
  - `git init`
    - 현재 폴더를 Git 저장소로 시작

- 상태 확인
  - `git status`
    - 지금 Git이 어떻게 보고 있는지 확인

- 변경 내용 확인
  - `git diff`
    - 아직 staging 안 된 변경 확인
  - `git diff --staged`
    - staging 된 변경 확인
  - 왜 중요하나
    - status는 "바뀌었다"까지만 보여줌
    - diff는 "정확히 뭐가 바뀌었는지" 보여줌

- staging
  - `git add`
    - commit에 포함할 변경 선택

- 저장
  - `git commit`
    - 현재 상태 기록
    - "의미 있는 변경 단위"를 남긴다고 설명

- 이력 확인
  - `git log`
    - 과거 기록 조회

- 핵심 메시지
  - "status로 보고, add로 고르고, commit으로 남긴다."
  - "diff로 내용 확인하는 습관이 중요하다."

- 멘트
  - "실무에서도 가장 많이 보는 건 status입니다."
  - "하지만 status만 보면 부족하고, 실제 변경 내용은 diff로 봐야 합니다."
  - "명령어보다 흐름을 먼저 기억하는 게 중요합니다."
  - "commit은 그냥 저장 버튼이 아니라, 나중에 다시 읽을 기록을 남기는 행위입니다."

## 3-0. Commit / Branch / Worktree를 나누는 기준

- 먼저 구분
  - commit = 기록 단위
  - branch = 작업 흐름 단위
  - worktree = 작업 공간 단위

- 같은 "작업 단위"처럼 보여도 레벨이 다름
  - branch 안에 여러 commit이 들어감
  - worktree는 보통 특정 branch 작업을 위한 실제 폴더 하나

- 쉽게 말하면
  - commit
    - 작업 일지 한 줄
  - branch
    - 작업 갈래 하나
  - worktree
    - 실제 책상 하나

- 판단 질문
  - commit
    - "이 변경을 한 줄 메시지로 설명할 수 있나?"
  - branch
    - "이 변경 흐름을 따로 리뷰하고 따로 합쳐야 하나?"
  - worktree
    - "이 작업을 다른 작업과 동시에 따로 열어둘 필요가 있나?"

- 핵심 멘트
  - "commit은 기록을 나누는 기준"
  - "branch는 개발 흐름을 나누는 기준"
  - "worktree는 작업 공간을 나누는 기준"

## 3-1. Commit message 작성 원칙

- 먼저 이해
  - commit message는 나중에 `log`에서 읽는 기록
  - 나 혼자 보기 위한 메모가 아니라
  - 미래의 나와 팀원이 읽는 설명

- 가장 중요한 원칙
  - 한 commit = 한 의미 단위
  - message는 그 변경의 핵심을 한 줄로 설명

- commit을 나누는 기준
  - 나중에 되돌리기 쉬운가
  - 리뷰하기 쉬운가
  - 메시지 한 줄로 설명 가능한가
  - unrelated한 변경이 섞이지 않았는가

- 좋은 commit의 기준
  - 무엇이 바뀌었는지 드러남
  - 왜 바꿨는지 유추 가능
  - 너무 크거나 애매한 범위를 한 commit에 몰지 않음

- 제목 작성 기준
  - 짧고 명확하게
  - 보통 한 줄 요약
  - 동사 중심으로 시작하면 읽기 쉬움
  - 예
    - add login form validation
    - fix null check in user service
    - refactor branch switching example

- 피해야 할 메시지
  - `update`
  - `fix`
  - `test`
  - `asdf`
  - 왜냐하면
    - 무엇을 바꿨는지 안 보임
    - 나중에 log에서 의미 추적이 어려움

- 제목에 들어가면 좋은 것
  - 무엇을
    - login, auth, branch, README
  - 어떻게
    - add, fix, remove, rename, refactor, docs

- 길어질 때 방법
  - 첫 줄은 요약
  - 필요하면 본문에서 이유 추가
  - 본문에 쓰면 좋은 내용
    - 왜 바꿨는지
    - 어떤 문제를 해결하는지
    - 주의할 점이 있는지

- 말투 원칙
  - 팀에서 한글 쓰면 한글로 통일
  - 팀에서 영어 쓰면 영어로 통일
  - 언어보다 일관성이 더 중요

- 입문자용 안전 기준
  - "무엇을 바꿨는가"는 반드시 쓰기
  - 가능하면 "왜 바꿨는가"도 드러나게 쓰기
  - 애매하면 파일명 나열보다 기능 단위로 쓰기

- 추천 설명
  - "좋은 commit message는 코드를 다시 열지 않아도 변경 목적이 대충 보여야 합니다."
  - "한 commit이 너무 크면 message도 애매해집니다. 결국 commit 단위부터 잘 나누는 게 중요합니다."

- 실무 선택지
  - 팀에 따라 Conventional Commits 사용 가능
    - `feat:`
    - `fix:`
    - `docs:`
    - `refactor:`
  - 하지만 입문 단계에서는 형식 암기보다
    - 의미 있는 단위
    - 명확한 제목
    - 일관성
    이 세 가지가 더 중요

## 3-2. `stash`

- 정의
  - 아직 commit 하기 애매한 변경을 잠깐 치워두는 기능
  - 작업 중이던 내용을 임시 서랍에 넣어두는 느낌

- 언제 쓰나
  - 급하게 브랜치 전환해야 할 때
  - pull이나 다른 작업을 먼저 해야 할 때
  - 아직 commit으로 남기고 싶지는 않을 때

- 기본 흐름
  - `git stash`
    - 현재 변경 임시 저장
  - `git stash list`
    - 저장된 stash 목록 보기
  - `git stash apply`
    - stash 다시 적용
  - `git stash pop`
    - 적용 후 stash 목록에서 제거

- 핵심 이해
  - commit은 공식 기록
  - stash는 임시 보관
  - 오래 쌓아두는 용도가 아님

- 입문자 주의
  - stash 했다고 영구 저장된 느낌으로 생각하지 않기
  - 여러 개 쌓이면 무엇을 넣었는지 헷갈릴 수 있음
  - 중요한 작업이면 stash보다 branch나 commit이 더 안전할 수 있음

- 멘트
  - "stash는 서랍에 잠깐 넣어두는 것"
  - "정식 기록이 필요하면 commit, 잠깐 치워둘 거면 stash"

## 4. Git vs 클라우드 스토리지

- 공통점
  - 파일 저장

- 차이점

| 항목 | Git | 클라우드 |
|---|---|---|
| 목적 | 버전 관리 | 파일 저장 |
| 이력 | 강력 | 제한적 |
| 협업 | 코드 중심 | 파일 중심 |
| 저장소 | 별도 repository 개념 | 서비스 자체에 포함 |

- 핵심 멘트
  - "Git은 도구이고, GitHub는 그 저장소와 협업 공간을 제공하는 서비스입니다."
  - "드라이브나 드롭박스처럼 저장만 하는 개념으로 보면 반만 이해한 겁니다."

## 5. Branch 개념

- 정의
  - 독립 작업 라인
  - 동일 저장소 안에서 분기

- 왜 필요한가
  - 기능 개발 분리
  - 실험 작업 분리
  - 안정 버전 보호

- branch를 나누는 기준
  - 따로 리뷰할 필요가 있는가
  - 따로 merge할 필요가 있는가
  - 다른 작업과 분리해서 진행해야 하는가
  - 보통 feature / bugfix / hotfix / experiment 단위

- 주요 개념
  - `main`
    - 기본 브랜치
  - `HEAD`
    - 현재 내가 보고 있는 위치
  - branch
    - "어느 흐름 위에서 작업 중인가"

- 핵심 메시지
  - "브랜치는 평행 세계"
  - "안전하게 분기해서 작업한 뒤 합친다"
  - "branch는 commit 하나가 아니라 commit들의 흐름을 나누는 것"

- 멘트
  - "main에서 다 같이 직접 작업하면 곧바로 충돌과 불안정이 생깁니다."
  - "실무에서는 branch 하나가 PR 하나와 대응되는 경우가 많습니다."

## 4-1. Branch 확장 개념 (Worktree 포함)

- 브랜치 복습
  - 브랜치 = 독립 작업 라인
  - 동일 저장소 내 분기

- 문제 상황
  - 브랜치 전환 시 파일 교체 발생
  - 동시에 여러 브랜치 작업 불편

- Worktree 등장
  - 여러 작업 디렉토리 생성
  - 각 디렉토리마다 다른 브랜치 연결

- 구조
  - 하나의 repository
  - 여러 working directory

- 핵심 개념
  - "브랜치는 논리적 분기"
  - "워크트리는 물리적 분리"

- 사용 이유
  - 병렬 작업
  - 충돌 방지
  - 빠른 컨텍스트 전환

- worktree를 나누는 기준
  - 동시에 두 작업을 열어둬야 하는가
  - branch를 자주 왔다 갔다 하면 불편한가
  - 빌드 결과나 작업 파일이 섞이면 곤란한가
  - 기능 개발 중인데 hotfix도 바로 봐야 하는 상황인가

- 예시 감각
  - 작은 버그 수정 1개
    - branch 1개
    - commit 1~3개
    - worktree 1개면 충분
  - 동시에 기능 개발 + 긴급 수정
    - branch 2개
    - worktree 2개가 편함

- 멘트
  - "요즘 바이브 코딩에서 자주 사용"
  - "여러 작업을 동시에 띄워놓고 진행"
  - "우리는 필수는 아니고, 개념만 이해하면 충분"
  - "worktree는 논리 분기가 아니라 물리적 작업 공간 분리입니다."

## 6. GitHub 개념

- GitHub 정의
  - Git 저장소 호스팅 서비스
  - 협업 플랫폼

- 역할
  - 원격 저장소 제공
  - 협업 기능 제공
  - 리뷰 / 권한 / 이슈 관리

- Git과의 관계
  - Git 자체 기능 아님
  - Git 위에 붙는 플랫폼

- 자주 나오는 용어
  - remote
    - 원격 저장소 연결 정보
  - `origin`
    - 가장 흔한 기본 remote 이름

- 핵심 멘트
  - "Git은 엔진, GitHub는 플랫폼"
  - "GitHub가 없어도 Git은 동작하지만, 협업은 GitHub 같은 플랫폼이 훨씬 편합니다."

## 7. Repository 연결 방법

- 방식 1: `clone`
  - 원격 → 로컬
  - 가장 단순
  - `.git`까지 포함해서 내려받음

- 방식 2: 새 프로젝트 연결
  - `git init`
  - `git remote add`
  - `git push`

- 방식 3: 기존 프로젝트 연결
  - 기존 코드 유지
  - remote 등록
  - 최초 push

- 핵심 강조
  - "`init + remote add + push` 흐름 중요"
  - "`clone`은 이미 완성된 저장소를 가져오는 것"

## 8. 협업 명령

- `fetch`
  - 원격 변경 가져오기
  - 내 브랜치에 바로 합치지는 않음

- `pull`
  - 원격 변경 가져오기
  - 보통 `fetch + merge` 또는 `fetch + rebase`까지 포함

- `branch`
  - 브랜치 생성 / 전환

- `merge`
  - 브랜치 통합

- `rebase`
  - 히스토리 정리

- 충돌
  - 같은 부분을 동시에 수정하면 발생
  - Git이 자동 판단 못 하면 사람 개입 필요

- 핵심 멘트
  - "fetch는 가져오기만"
  - "pull은 가져오고 합치기까지"
  - "merge는 합치기"
  - "rebase는 정리"
  - "충돌은 이상 현상이 아니라 협업 과정의 일부"

## 8-1. `fetch` vs `pull`

- `fetch`
  - 원격 저장소 최신 상태를 로컬로 가져오기만 함
  - 내 작업 파일은 바로 바뀌지 않음
  - 안전하게 상황 파악할 때 유리

- `pull`
  - 가져온 뒤 현재 브랜치에 반영까지 진행
  - 편하지만 상황을 모르고 쓰면 갑자기 merge / rebase가 일어날 수 있음

- 입문자 권장 감각
  - "불안하면 fetch 먼저"
  - "상태 보고, diff 보고, 그 다음 merge나 pull"

- 예시 흐름
  - 원격 상태 확인 필요
  - `fetch`
  - `log`, `diff`, 브랜치 상태 확인
  - 필요 시 `merge` 또는 `pull`

- 멘트
  - "pull은 편하지만 자동 동작이 더 들어간 명령입니다."
  - "입문자는 fetch와 pull의 차이를 꼭 알고 있어야 합니다."


---

## 8-2. 충돌 났을 때 최소 대응 흐름

- 먼저 이해
  - 충돌 = 망가짐이 아님
  - Git이 어느 쪽을 남길지 자동 결정 못 한 상태

- 왜 생기나
  - 같은 파일
  - 같은 줄 주변
  - 서로 다른 변경이 겹침

- 최소 대응 순서
  - 어떤 파일에서 충돌 났는지 확인
  - 충돌 표시 구간 확인
  - 어느 내용을 남길지 결정
  - 필요하면 두 내용 섞어서 정리
  - 충돌 표시 제거
  - `add`
  - `commit` 또는 merge 완료

- 입문자 주의
  - 당황해서 파일 전체를 지우지 않기
  - 잘 모르면 바로 덮어쓰기하지 않기
  - 충돌 해결 후 diff 다시 보기

- 핵심 멘트
  - "충돌은 오류가 아니라 의사결정 요청"
  - "해결 후 add 해야 Git이 해결된 것으로 인식"

## 9-1. `revert` vs `reset`

- 공통점
  - 둘 다 이전 변경을 되돌리는 데 사용

- `revert`
  - 기존 히스토리는 유지
  - "되돌리는 commit"을 새로 하나 추가
  - 협업 브랜치에서 상대적으로 안전

- `reset`
  - 현재 브랜치의 위치 자체를 옮김
  - local 히스토리를 바꿀 수 있음
  - 강하게 쓰면 작업 내용이 사라질 수 있음

- 감각적으로 구분
  - `revert`
    - 기록을 남기면서 취소
  - `reset`
    - 시간을 되돌려 위치를 옮김

- 언제 무엇을 쓰나
  - 이미 공유된 commit 취소
    - `revert` 쪽이 일반적으로 안전
  - 아직 혼자 작업 중인 local 정리
    - `reset`이 유용할 수 있음

- 입문자 주의
  - `reset --hard`는 매우 조심
  - 공유 브랜치에서 히스토리 재작성은 위험
  - 모르겠으면 reset보다 revert 우선

- 핵심 멘트
  - "협업에서는 revert가 더 안전한 기본값"
  - "reset은 강력하지만 그만큼 위험할 수 있다"

## 16-1. AI와 함께 Git 사용할 때 안전수칙

- 기본 원칙
  - AI가 명령어를 제안해도 그대로 무의식적으로 실행하지 않기
  - 현재 상태를 먼저 확인하고 실행

- 실행 전 최소 확인
  - `git status`
    - 지금 변경 상태 확인
  - `git diff`
    - 실제 변경 내용 확인
  - `git log --oneline`
    - 현재 브랜치 최근 기록 확인

- 공유 작업 전 확인
  - 지금 브랜치가 어디인지 확인
  - 내 작업이 `main`인지 feature branch인지 확인
  - push 전에 diff / commit 메시지 재확인

- AI에게 물어볼 때 좋은 질문 방식
  - "현재 status 기준으로 다음 안전한 명령 알려줘"
  - "이 상황에서 reset 말고 안전한 방법 있나?"
  - "push 전에 확인할 것 체크리스트 줘"

- 특히 조심할 명령
  - `reset --hard`
  - 강제 push
  - 무의식적인 rebase
  - 충돌 난 상태에서 대충 덮어쓰기

- 추천 습관
  - 중요한 작업 전 branch 확인
  - pull 전에 작업 중인 변경 있는지 확인
  - 이상하면 fetch 먼저
  - commit 전 diff 한 번 더 보기

- 핵심 멘트
  - "AI는 보조자이고, 상태 판단은 내가 해야 합니다."
  - "Git은 명령어보다 현재 상태 확인이 먼저입니다."
