# CPU 구조

## 프로그램의 실행

```text
SW Coding -> Compiler -> Assembly/Object file -> Linker -> Executable -> 실행 코드(머신코드)
```

- 소스 코드는 컴파일 과정을 거쳐 머신 코드로 변환된다
- 변환된 코드는 메모리에 저장되고 CPU가 instruction 0, 1, 2, 3, ... 순서로 읽어 실행
- 예:
  - `A = 0;`
  - `A = A + 1;`
  - `out = A;`
- 이런 코드도 결국은 여러 개의 instruction으로 쪼개져 순차 실행
`
## CPU 기본 구조와 메모리 아키텍처

### 폰 노이만 구조 (Von Neumann Architecture)

```text
CPU <-> Memory
```

- 데이터와 명령어를 하나의 메모리에 함께 저장
- 하나의 메모리/버스를 공유해서 처리
- CPU가 명령어를 가져오고, 메모리에서 데이터를 읽고/쓰는 과정에서 병목 발생 가능

### 하버드 구조 (Harvard Architecture)

```text
Program Memory(ROM/Flash) -> CPU <-> Data Memory(RAM)
```

- 명령어 메모리와 데이터 메모리를 물리적으로 분리
- 명령어와 데이터를 동시에 접근 가능
- 성능에 유리
- 현재 MCU/임베디드 구조에서는 하버드 구조 또는 그 응용 구조가 많이 사용된다

# CISC와 RISC

## ISA(Instruction Set Architecture)

- ISA = CPU가 어떤 명령어를 이해하고 어떤 방식으로 동작해야 하는지를 정의한 규격
- CPU 아키텍처는 ISA 설계 철학에 따라 크게 CISC와 RISC로 나눌 수 있음

## CISC (Complex Instruction Set Computer)

- 개념: 복잡하고 종류가 다양한 명령어 셋을 갖는 구조
- 배경: 과거 메모리 용량이 작았기 때문에, 프로그램 크기를 줄이기 위해 하나의 명령어로 여러 작업 수행
- 대표 예: Intel x86 계열

장점:

* 복합 기능 제공
* 하위 호환성 우수
* 범용 PC 환경에 적합

단점:

* 로직 복잡
* 트랜지스터 효율 저하 가능
* 전력 소모 큼
* 속도 느려질 수 있음
* 가격 상승 가능

용도:

* PC 등 범용 컴퓨터

## RISC (Reduced Instruction Set Computer)

- 개념: 사용 빈도가 높은 핵심 명령어 위주로 단순화한 구조
- 핵심 아이디어: CISC 명령어 중 자주 쓰는 일부 명령어가 대부분의 일을 처리한다는 관점
- 흔한 설명: 자주 쓰는 20% 정도의 명령어가 전체 일의 80% 이상을 처리
- 대표 예: ARM, MIPS

장점:

* 로직 단순
* 고속 동작에 유리
* 전력 소모와 발열 감소에 유리
* 가격 절감 가능
* 파이프라인 처리에 유리

단점:

* 소프트웨어가 커지고 복잡해질 수 있음
* 하위 호환성이 부족할 수 있음
* 여러 개의 단순한 명령어 조합이 필요

용도:

* 임베디드 프로세서
* 전력/발열 효율이 중요한 시스템

## ARM / RISC-V / MIPS / ARC

- ARM
  - RISC 계열 ISA
  - ARM Ltd.가 설계하고 라이선스하는 구조
  - 오픈 ISA는 아님
  - 상용화 시 라이선스 비용/로열티 구조 부담 가능
- RISC-V
  - RISC 계열 오픈 ISA
  - ISA 자체 사용에는 로열티가 없음
  - ARM 대안으로 자주 언급된다
  - 구현체를 반드시 공개해야 하는 구조는 아님
  - 아직 ARM만큼 ecosystem이 성숙했다고 보기는 어려움
- MIPS / ARC
  - 모두 RISC 계열
  - 현재는 ARM / RISC-V 대비 영향력 감소
- 인물 메모
  - David Patterson = UC Berkeley RISC 흐름 핵심 인물
  - John Hennessy = Stanford / MIPS 흐름 핵심 인물

## RISC-V 개요

- 정의: 2010년 미국 UC Berkeley에서 시작된 오픈 RISC ISA
- 이름: Berkeley 쪽 RISC 흐름에서 이어진 5번째 RISC ISA라는 의미로 이해 가능
- 특징:
  - 개방형 ISA
  - 로열티 없음
  - 설계 자유도 높음
  - ARM이나 AMD64와 대비되는 개방형 성격
  - 학술/연구용에서 시작했지만 MCU/내장 제어 프로세서 산업 전반으로 확산 중
- 규격:
  - 32비트 / 64비트 / 128비트 지원
  - 현재 CC-BY 4.0 라이선스로 배포
- 주요 확장 집합:
  - RV32I = 기본 정수 명령어 집합
  - RV32M = 정수 곱셈/나눗셈 확장
  - RV32A = Atomic 명령어 확장
  - RV32C = Compressed 명령어 확장

**한 줄 정리**
- CISC: 복잡한 명령어 기반, 호환성 강점
- RISC: 단순한 핵심 명령어 기반, 파이프라인/전력 효율 강점

비트폭 메모:

- 16비트, 32비트, 64비트 같은 표현은 instruction 길이와 데이터 처리 폭, ALU가 한 번에 처리하는 데이터 길이와 연결

# RISC-V로 CPU 구조와 동작 설명하기

## 왜 RISC-V를 예시로 쓰는가

- CPU 구조와 동작을 설명할 때는 RISC-V를 예시 ISA로 잡으면 충분
- instruction decode, register, pipeline, branch, memory hierarchy 같은 핵심 개념을 설명하기 좋음
- ARM이나 x86보다 이름이 덜 익숙해서, 처음 듣는 사람은 CPU 이름인지 규격 이름인지부터 헷갈릴 수 있다
- 그래서 본론 전에 "RISC-V가 무엇인지"를 먼저 설명하고 들어가는 편이 좋음

## 먼저 알아둘 것

| 항목             | 의미                    |
| -------------- | --------------------- |
| RISC-V         | CPU 설계 규격인 ISA        |
| HDL            | CPU 회로 구현 언어          |
| RISC-V 공식 repo | 문서/명세/테스트/툴 중심 저장소   |
| RISC-V CPU core | HDL로 구현된 실제 프로세서     |

따라서

```text
"RISC-V를 HDL로 구현한다"
```

는 정확한 표현이고,

Computer Architecture에서 말하는 **RISC-V** = HDL이 아니라 **ISA(Instruction Set Architecture)**.

RISC-V가 정의하는 것:

* 명령어 형식
* 레지스터 구조
* 메모리 접근 방식
* 예외 처리 규칙
* CPU가 해야 하는 동작 규칙

HDL이 구현하는 것:

* 실제 회로 구조
* 파이프라인
* ALU
* 레지스터 파일
* 캐시
* 버스 연결

대표적인 HDL / 하드웨어 기술 언어 계열:

* Verilog
* SystemVerilog
* VHDL
* Chisel
* SpinalHDL

즉,

```text
RISC-V = 규칙
CPU core = 그 규칙을 구현한 회로
```

로 이해하면 된다.

## RISC-V로 아키텍처를 배운다는 뜻

### 1. ISA 학습 관점

- instruction decode
- register
- pipeline
- branch
- memory hierarchy

즉, ISA 관점의 이론 학습.

### 2. HDL 구현 관점

- 같은 RISC-V spec을 보고 만들어도, 내부 datapath / control / microarchitecture는 구현마다 달라질 수 있음
- 성능 개선 포인트도 구현자가 직접 잡아야 하는 부분

수업 포커싱:

- 가장 기본: single-cycle 처리 CPU
- 다음 확장: multi-cycle CPU
- 다음 확장: pipeline 도입 CPU
- single-cycle, multi-cycle까지는 핵심 범위
- pipeline은 난이도가 높아서 선택적 심화 범위에 가까움
- 발표는 single-cycle CPU 기준
- 팀플은 multi-cycle + AMBA bus + peripheral까지 가는 방향
- multi-cycle로 가면 소프트웨어 동작 과정과 제어 흐름 해석도 달라짐
- 책을 보고 pipeline까지 개인적으로 더 확장하는 것은 개인 진행 영역

## ISA와 HDL의 계층

ISA와 HDL은 계층이 다름.

```text
[ RISC-V ISA ]
        ↓
[ CPU Microarchitecture ]
        ↓
[ HDL Implementation ]
        ↓
[ FPGA / ASIC ]
```

- ISA = 무엇을 해야 하는가
- Microarchitecture = 그 규칙을 어떤 구조로 구현할 것인가
- HDL = 그 구조를 실제 회로 코드로 어떻게 표현할 것인가

예:

- ARM ISA를 Verilog로 구현 가능
- x86 ISA를 Verilog로 구현 가능
- RISC-V ISA를 Verilog로 구현 가능

### 구조 설명

해당 구조는 CPU 중심의 SoC(System on Chip) 구조를 단순화한 형태.

구성 요소별 역할:

### 1. Instruction Memory (ROM)

CPU가 실행할 명령어 저장.

주요 특징:

* 비휘발성 메모리 사용
* 프로그램 코드 저장
* CPU가 Address 전달
* ROM이 Instruction 반환

동작 흐름:

```text
CPU -> Address
ROM -> Instruction
```

일반적으로 Flash ROM 또는 Boot ROM 사용.

### 2. CPU

CPU는 크게 `Data Path`와 `Control Unit`으로 구성.

#### 2.1 Data Path

실제 연산 수행 블록.

- CPU 설계에서 중심은 `Control Unit`보다 `Data Path`
- Control Unit은 Data Path를 제어하는 역할
- 실제 CPU 구조/동작을 이해할 때는 Data Path를 중심으로 보는 것이 중요

구성 예시:

* ALU
* Register File
* Program Counter
* Multiplexer
* Shifter

역할:

* 산술 연산 수행
* 데이터 이동
* 메모리 접근
* 결과 저장

핵심 포인트:

* 메모리 계층 중 CPU 내부에서 가장 빠른 저장소는 레지스터
* 레지스터는 명령어, 주소, 연산 데이터 같은 CPU 내부 핵심 상태를 매 cycle 단위로 저장/전달
* Data Path의 핵심은 ALU만이 아니라 레지스터 흐름
* ALU는 레지스터에 들어 있는 값을 가져와 연산하는 블록
* 따라서 Data Path 설계에서는 "어떤 값을 어떤 레지스터에 저장하고, 언제 ALU로 보내고, 결과를 어디에 다시 저장하는가"가 핵심

#### 2.2 Control Unit

Data Path 제어 수행 블록.

역할:

* 명령어 해석
* 제어 신호 생성
* 상태 전이 관리
* 버스 접근 제어

생성 예시:

* Read Enable
* Write Enable
* ALU Op
* Register Write
* Memory Access Control

##### 전체 구조 개요

전용 CPU(Dedicated CPU)는 특정 연산/제어 흐름만 수행하도록 만든 CPU 구조로 볼 수 있음.

구조는 다음 두 영역으로 분리된다.

| 영역           | 역할           |
| ------------ | ------------ |
| Datapath     | 실제 데이터 연산 수행 |
| Control Unit | Datapath 제어  |

여기서는 가장 단순한 dedicated CPU 예시로

```text
A = 0
A = A + 1
out = A
```

흐름을 먼저 이해함.

실제 구현에서는 `eq9` 같은 종료 조건 비교 신호를 추가해서 반복 제어까지 포함함.

##### 1. Datapath 설명

Datapath는 데이터 흐름과 연산 수행 담당.

##### 1.1 MUX

입력 선택기.

```text
A_src_sel
```

신호에 따라 입력 선택.

입력 후보:

| 입력             | 설명     |
| -------------- | ------ |
| constant 0     | 초기값    |
| feedback path  | ALU 결과 |

동작:

- `A_src_sel = 0` : constant 0 선택, 초기화 수행
- `A_src_sel = 1` : ALU 결과 선택, 누산(accumulation) 수행

##### 1.2 A_Reg

Accumulator register 역할 수행.

기능:

* 데이터 저장
* clock edge에서 값 갱신
* load enable 지원

제어 신호:

| 신호         | 역할             |
| ---------- | -------------- |
| A_Reg_load | Register 갱신 허용 |

동작:

- `A_Reg_load = 1` : 다음 clock edge에서 값 저장
- `A_Reg_load = 0` : 현재 값 유지

##### 1.3 ALU

산술 연산 수행.

현재 구조에서는

```text
A + 1
```

수행.

입력:

| 입력 | 설명         |
| -- | ---------- |
| a  | A_Reg 값    |
| b  | constant 1 |

출력:

```text
A + 1
```

##### 1.4 Comparator (`eq9`)

실제 구현에서는 comparator가 추가되어 `A_Reg` 값이 특정 값에 도달했는지 비교함.

역할:

* 종료 조건 신호 생성
* Control Unit의 상태 전이 조건 제공

예:

```text
eq9 = (A_Reg == 9)
```

동작:

- `eq9 = 0` : 아직 반복 계속
- `eq9 = 1` : 목표 count 도달, 다음 단계에서 출력 상태로 전이

##### 1.5 Feedback Path

ALU 결과를 다시 입력으로 전달.

핵심 역할:

```text
누산 연산 형성
```

즉:

```text
A(n+1) = A(n) + 1
```

구조 생성.

##### 1.6 Tri-state Buffer

출력 버퍼 역할 수행.

제어 신호:

```text
out_sel
```

동작:

- `out_sel = 1` : 출력 활성화
- `out_sel = 0` : High-Z 상태, 출력 비활성화

##### 2. Control Unit 설명

Control Unit은 FSM(Finite State Machine) 형태로 동작.

각 state에서 제어 신호 생성.

##### 2.1 State S0

초기화 단계.

```text
A = 0
```

제어값:

```text
A_src_sel = 0
A_Reg_load = 1
out_sel = 0
```

의미:

| 제어           | 의미          |
| ------------ | ----------- |
| A_src_sel=0  | constant 0 선택 |
| A_Reg_load=1 | Register 저장 |
| out_sel=0    | 출력 비활성      |

결과:

```text
A_Reg <- 0
```

##### 2.2 State S1

증가 연산 수행.

```text
A = A + 1
```

제어값:

```text
A_src_sel = 1
A_Reg_load = 1
out_sel = 0
```

의미:

| 제어           | 의미          |
| ------------ | ----------- |
| A_src_sel=1  | Feedback 선택 |
| A_Reg_load=1 | 결과 저장       |
| out_sel=0    | 출력 비활성      |

결과:

```text
A_Reg <- A_Reg + 1
```

보충:

- `eq9 = 0` 이면 S1 상태 유지, 계속 증가
- `eq9 = 1` 이면 다음 clock에서 S2로 전이

##### 2.3 State S2

최종 출력 단계.

```text
out = A
```

제어값:

```text
A_Reg_load = 0
out_sel = 1
```

의미:

| 제어           | 의미    |
| ------------ | ----- |
| A_Reg_load=0 | 값 유지  |
| out_sel=1    | 출력 활성 |

결과:

```text
out <- A_Reg
```

##### 3. Clock 기준 동작 흐름

Reset 직후 / State S0:

```text
A_Reg <- 0
```

결과:

```text
A_Reg = 0
```

State S1 반복:

```text
A_Reg = 01 -> 02 -> ... -> 09
```

특징:

* `eq9 = 0` 인 동안 S1 유지
* `alu_result`는 현재 `A_Reg`보다 항상 1 큰 값 유지

상태 전이:

```text
A_Reg = 09 가 되면 eq9 = 1
다음 clock에서 S2로 전이
```

State S2:

```text
out_sel = 1
최종 결과 출력
```

시뮬레이션 예시에서는 최종 결과값이 `0a`, 즉 `10`으로 보일 수 있음.

##### 4. 구조적 특징

특징:

* FSM 기반 제어
* Register 중심 구조
* Feedback 기반 누산
* 순차회로 구성
* Clock 동기식 설계

##### 5. RTL 관점 해석

구조는 다음 RTL과 유사.

```verilog
always @(posedge clk) begin
    if (A_Reg_load)
        A_Reg <= mux_out;
end

assign alu_out = A_Reg + 1;

assign mux_out =
    (A_src_sel == 0) ? 0 :
                       alu_out;

assign out =
    (out_sel) ? A_Reg : 'bz;
```

##### 6. 설계 관점 의미

이 구조는 디지털 설계의 핵심 개념 포함.

포함 개념:

* Datapath / Control 분리
* FSM 제어
* Register Transfer
* Feedback Loop
* Sequential Logic
* ALU 기반 연산
* MUX 기반 경로 선택

CPU 내부 구조의 축소 모델로 볼 수 있음.

##### 7. 실습 과제: 0부터 10까지의 합 계산

목표:

```text
0 + 1 + 2 + ... + 10 = 55
```

알고리즘 관점:

```c
sum = 0;
a = 0;

while (a < 10) {
    a = a + 1;
    sum = sum + a;
}

out = sum;
halt;
```

의미:

* 단순 counter를 넘어서 register가 여러 개 필요한 구조로 확장된다
* `a`와 `sum`을 각각 저장할 register 필요
* comparator 조건도 `a < 10` 형태로 확장된다
* dedicated CPU도 알고리즘이 복잡해질수록 datapath와 control이 같이 커짐

##### 8. Verilog HDL 구현 관점

`data_path` 모듈:

* `mux_2x1` : 입력 선택
* `a_reg` : clock 동기 저장
* `alu` : 8비트 덧셈 수행
* `comp_eq9` : `eq9` 생성
* `tri-state output` 또는 output select 경로 : 최종 출력 제어

`control_unit` 모듈:

* `typedef enum` 으로 state 정의
* `always_ff` : reset / 현재 state 저장
* `always_comb` : 다음 state 및 제어 신호 생성

핵심 구조는 다음과 같다.

* Datapath는 실제 연산 수행
* Control Unit은 state에 따라 `asrc_sel`, `areg_load`, `out_sel` 같은 제어 신호 생성

##### 9. 시뮬레이션 / 검증 포인트

검증 포인트는 다음과 같다.

* reset 직후 S0에서 `A_Reg`가 0으로 초기화되는가
* S1에서 `A_Reg` 값이 `01 -> 02 -> ... -> 09`로 증가하는가
* `A_Reg = 09` 시점에 `eq9 = 1`이 되는가
* `c_state`가 S1에 머물며 반복하다가 조건 만족 시 S2로 전이하는가
* `alu_result`가 항상 현재 register 값보다 1 큰가
* S2에서 `out_sel`이 활성화되고 최종 결과가 `out`으로 나가는가

정리하면 다음과 같다.

* Dedicated CPU는 Datapath와 Control Unit으로 나뉘어 설계
* 제어 유닛은 FSM으로 각 단계 수행
* HDL로 구현 후 시뮬레이션으로 상태 전이와 데이터 흐름 검증 가능

### 3. Data Memory (RAM)

런타임 데이터 저장.

* Stack

| 신호         | 설명            |

### 4. AMBA Bus

CPU와 메모리/주변장치를 연결하는 인터커넥트 구조.

AMBA는 ARM에서 정의한 버스 프로토콜.

사용 가능한 버스:

| 버스  | 특징               |
| --- | ---------------- |
| AHB | 고속 시스템 버스        |
| APB | 저속 Peripheral 버스 |

### 5. AHB (Advanced High-performance Bus)

고성능 데이터 전송용.

주 사용 대상:

* CPU
* RAM
* DMA
* High-Speed Peripheral

특징:

* 파이프라인 지원
* Burst 전송 지원
* 고속 처리 가능
* 높은 대역폭 제공

적합한 장치:

* SRAM
* DRAM Controller
* Ethernet
* USB

### 6. APB (Advanced Peripheral Bus)

저속 Peripheral 연결용.

특징:

* 단순 프로토콜
* 저전력
* 낮은 구현 복잡도
* 비파이프라인 구조

적합한 장치:

* UART
* GPIO
* TIMER
* Watchdog
* RTC

### 7. Peripheral

메모리 맵 방식으로 CPU에 연결.

CPU는 Peripheral을 메모리처럼 접근.

예시:

| 주소          | 장치    |
| ----------- | ----- |
| 0x4000_0000 | GPIO  |
| 0x4000_1000 | UART  |
| 0x4000_2000 | TIMER |

접근 예시:

```c
*(volatile uint32_t*)0x40001000 = 0x55;
```

위 코드는 UART register 접근 예시.

### 8. 전체 데이터 흐름

명령어 Fetch:

```text
CPU -> ROM Address 전달
ROM -> Instruction 반환
CPU -> 명령어 실행
```

데이터 Read:

```text
CPU -> Bus -> RAM Address 전달
RAM -> Read Data 반환
CPU -> Register 저장
```

Peripheral Write:

```text
CPU -> AMBA Bus
Bus -> Peripheral 선택
Peripheral Register Write 수행
```

### 9. Bus 기반 구조의 핵심 특징

장점:

* 모듈 확장 용이

* 표준 인터페이스 사용

단점:

* Bus 병목 가능

* 대역폭 제한 존재

# RISC-V는 문서인가?

## 핵심

핵심적으로는 맞다.

RISC-V의 본질은

> CPU가 따라야 하는 규칙을 정의한 명세(specification)

로 볼 수 있음.

즉 RISC-V 자체는 아래 자체를 가리키는 말이 아님.

- CPU 회로
- 칩
- HDL 코드

반대로 아래를 정의하는 ISA specification에 가까움.

- 명령어 규칙
- 동작 규칙
- 인터페이스 규칙

## 예시

RISC-V 문서에는 아래와 같은 정의가 들어감.

```text
ADD rd, rs1, rs2
```

의 의미:

```text
rd = rs1 + rs2
```

instruction encoding:

```text
funct7 | rs2 | rs1 | funct3 | rd | opcode
```

같은 비트 배치도 함께 정의함.

하지만 아래 내용은 정의하지 않음.

- ALU를 몇 단계로 만들지
- pipeline을 몇 stage로 할지
- cache를 넣을지
- branch predictor를 넣을지

이건 구현자의 자유임.

## 그래서 구현체가 다양함

같은 RISC-V ISA라도 아래처럼 다양한 구현 가능.

| 구현체            | 특징                  |
| -------------- | ------------------- |
| 작은 MCU         | 단순 single-cycle     |
| FPGA 학습용       | 5-stage pipeline    |
| 서버 CPU         | superscalar + OoO   |
| AI accelerator | vector extension 포함 |

ISA만 같으면 소프트웨어 호환성 유지 가능.

## 비유

RISC-V는 "언어 문법"에 가까움.

```text
영어 문법
```

은 정의 문서이고,

- 사람이 말할 수도 있고
- AI가 말할 수도 있고
- 로봇이 말할 수도 있다

구현은 달라도 문법은 같다.

마찬가지로

```text
RISC-V ISA
```

는 CPU 동작 문법이고,
실제 CPU는 그 문법을 구현한 결과물임.

## 실제 오픈소스 CPU는 따로 존재

아래 저장소들은 실제 HDL 구현체임.

## RISC / MIPS / RISC-V 관계

- RISC는 명령어 집합을 단순화하는 설계 철학
- MIPS는 John Hennessy와 Stanford 쪽 흐름에 강하게 연결되는 초기 RISC 계열의 대표 아키텍처
- David Patterson은 UC Berkeley에서 RISC 정립을 이끈 핵심 인물
- RISC-V는 UC Berkeley에서 시작한 오픈 ISA
- RISC-V는 같은 RISC 계열이지만, 오픈 ISA라는 점이 핵심 차이

## 정리

| 항목             | 의미                       |
| -------------- | ------------------------ |
| RISC            | 명령어 집합을 단순화하는 설계 철학      |
| RISC-V          | ISA specification        |
| RISC-V 공식 repo | 문서/툴/테스트 중심              |
| HDL 구현체        | 실제 CPU 회로 코드             |
| TeX            | ISA 문서를 만들기 위한 기술 문서 시스템 |

```text
RISC-V = CPU 설계 규칙 문서
HDL 구현체 = 그 규칙을 실제 회로로 만든 코드
```

# Dedicated CPU 누적합 설계 과제

### 설계 목표

0부터 10까지의 합을 계산하는 Dedicated CPU 설계.

```text
0 + 1 + 2 + 3 + ... + 9 + 10 = 55
```

설계 내용:

* 해당 동작을 수행하는 코드 작성
* B/D(Block Diagram) 기반 시뮬레이션 또는 결과 정리

### 알고리즘 / 상태 설계

초기화:

```text
S0: a = 0;
S1: sum = 0;
```

반복 / 조건 판단:

```text
S2: while (a < 10) 조건 확인
S3: a = a + 1;
S4: sum = sum + a;
```

종료 / 출력:

```text
S5: out = sum;
    halt;
```

의미:

* `a`는 1부터 10까지 세는 값
* `sum`은 누적 합 저장 값
* 최종 `out`은 `a`가 아니라 `sum`을 출력
* 최종 결과는 `55`

### 설계 참고 사항

앞에서 다룬 Dedicated CPU Counter 구조를 확장해서 설계한다.

중요한 차이는 다음과 같다.

* counter 예제는 register를 1개만 사용
* 누적합 설계는 register를 2개 사용해야 한다
* 즉 `A_REG` 하나만으로 끝나는 구조가 아니라 `A_REG + SUM_REG` 구조로 확장해야 한다

Datapath:

* `A_REG` : counter용 register
* `SUM_REG` : 누적 합 저장 register
* ALU(덧셈기)
* MUX(입력 선택)
* 비교기(condition check)

데이터 흐름은 다음과 같다.

* `A_REG`는 1, 2, 3, ... , 10 쪽 흐름 담당
* `SUM_REG`는 1, 3, 6, 10, ... , 55 쪽 흐름 담당

Control Unit:

* state별 제어 신호 생성
* `AREG_Load`, `SumREG_Load` 같은 register load signal 필요
* 필요하면 `asrc_sel`, `sum_src_sel`, `out_sel` 같은 control signal 추가
* FSM 기반 상태 전이 설계

입출력:

* `clk`, `rst` 입력
* `out[7:0]` 출력
* `out`은 최종적으로 `SUM_REG` 값, 즉 `55`를 출력한다

핵심 구조는 다음과 같다.

* 전용 CPU 설계 원리를 그대로 확장하는 과제
* counter 예제는 register 1개짜리 입문 예제
* 누적합 설계는 register 2개를 쓰는 확장형 dedicated CPU 설계
