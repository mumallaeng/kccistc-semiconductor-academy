# 26-06-09 - UVM

0609 수업은 UVM(Universal Verification Methodology) 기반 검증 환경 구성과 Makefile을 이용한 시뮬레이션 자동화 방법을 중심으로 진행됐다.

### UVM 검증 환경 구조

* **핵심 컴포넌트**: `uvm_driver`, `uvm_sequencer`, `uvm_monitor`, `uvm_scoreboard`, `uvm_agent`, `uvm_env` 등이 포함됩니다.
* **데이터 흐름**: 트랜잭션(`uvm_sequence_item`)은 시퀀스에서 생성되어 시퀀서를 거쳐 드라이버가 DUT에 전달하며, 모니터가 이를 감지하여 스코어보드나 서브스크라이버로 전송합니다.
* **파라미터화(`#(type)`)**:
  * `uvm_sequence`, `uvm_sequencer`, `uvm_driver`, `uvm_subscriber`와 같이 데이터 전송 파이프라인에 직접 관여하는 클래스는 타입 파라미터를 사용하여 데이터 형식을 정의합니다.
  * 반면, `uvm_monitor`, `uvm_agent`, `uvm_env`, `uvm_scoreboard`는 여러 패킷 타입을 다루거나 컨테이너 역할을 하므로 일반적으로 클래스 레벨의 타입 파라미터를 사용하지 않습니다.

### Makefile을 이용한 자동화

Makefile은 컴파일 및 시뮬레이션 과정을 자동화하며, 주요 규칙은 다음과 같습니다.

* **기본 구조**: `타겟: 의존성` 형태를 가지며, 레시피 앞에는 반드시 `<TAB>`을 사용해야 합니다.
* **VCS 명령어**: `vcs`를 통해 컴파일하고 `simv`를 실행하여 검증을 수행합니다.
* **변수 활용**: 가독성과 유지보수를 위해 `VCS_OPTS`, `SIMV_OPTS` 등의 변수를 정의하여 사용합니다.
* **주요 문법**:
  * `.PHONY`: 타겟 이름과 파일 이름이 동일할 경우 타겟을 우선 실행하도록 지정합니다.
  * `$@`: 현재 타겟 이름.
  * `$<`: 첫 번째 의존성 이름.
  * `@`: 명령어 자체를 터미널에 출력하지 않게 합니다.

UVM 구조와 Makefile 실행 흐름은 이후 RAM 실습으로 확장할 때도 그대로 재사용되는 기준이다.

### 1\. UVM 기초 및 객체 구조

* **uvm\_object**: Stimulus, 데이터, test bench 입력값 등을 정의하는 가장 기본적인 객체임. `copy()`, `compare()`, `print()` 등의 기본 메서드를 포함함.
* **uvm\_transaction**: 데이터 전송을 위한 기본 클래스이며, 이를 상속받은 `uvm_sequence_item`이 실제 전송 단위가 됨.
* **uvm\_report\_object**: 메시지 로깅 기능을 제공하며 `uvm_report_*()` 함수를 통해 로그를 남김.
* **uvm\_component**: 검증 환경을 구성하는 계층적 구조의 객체임.
  * **uvm\_test**: 검증 시나리오를 설정하고 환경을 빌드함.
  * **uvm\_env**: agent, scoreboard 등 하위 컴포넌트들을 담는 컨테이너 역할을 함.
  * **uvm\_agent**: sequencer, driver, monitor를 하나로 묶어 관리하는 단위임.
  * **uvm\_sequencer**: sequence로부터 item을 받아 driver로 중계하며 중재(Arbitration) 기능을 수행함.
  * **uvm\_driver**: sequencer로부터 item을 받아 인터페이스를 통해 DUT에 신호를 인가함.
  * **uvm\_monitor**: 인터페이스의 신호를 관찰하여 트랜잭션 단위로 변환하고 분석 포트로 전송함.
  * **uvm\_scoreboard**: monitor로부터 받은 데이터를 예상 결과와 비교하여 검증을 수행함.
  * **uvm\_subscriber**: 분석 포트로부터 데이터를 받아 커버리지 측정 등에 활용함.

### 2\. TLM (Transaction Level Modeling) 통신

* **Port & Export**: 발신 측은 `port`를, 수신 측은 `export` 또는 `imp`를 사용하여 연결함.
* **Analysis Port**: 1:N 통신을 지원하며, 수신 측에서 구현한 `write()` 기능을 호출하여 데이터를 전송함.
* **연결 구조**:
  * `driver.seq_item_port.connect(sequencer.seq_item_export)`.
  * `monitor.ap.connect(scoreboard.ap_imp)`.

### 3\. 클래스 파라미터 `#(type)` 사용 규칙

* **사용하는 클래스**: 시퀀스 아이템(패킷)을 직접 주고받는 핵심 파이프라인 컴포넌트들임.
  * `uvm_sequence`, `uvm_sequencer`, `uvm_driver`, `uvm_subscriber`가 해당함.
  * 이유: 내장된 핸들러나 포트의 데이터 타입을 결정하여 캐스팅 없이 멤버 변수에 접근하기 위함임.
* **사용하지 않는 클래스**: 특정 트랜잭션에 고정되지 않는 컨테이너 성격의 컴포넌트들임.
  * `uvm_monitor`, `uvm_agent`, `uvm_env`, `uvm_scoreboard` 등이 해당함.
  * 이유: 여러 종류의 패킷을 동시에 다루거나 하위 컴포넌트를 담는 역할에 집중하기 때문임.

### 4\. Makefile 작성 및 활용

* **기본 문법**: `target: dependency <TAB> recipe` 구조임.
* **변수 할당 방식**:
  * `=`: 재귀 확장. 참조 시점에 값이 결정됨.
  * `:=`: 즉시 확장. 선언 시점에 값이 고정됨.
  * `?=`: 조건부 할당. 변수가 미정의된 경우에만 할당함.
  * `+=`: 값 추가. 기존 값 뒤에 공백과 함께 추가함.
* **주요 자동 변수**:
  * `$@`: 현재 타겟의 이름.
  * `$<`: 첫 번째 의존성(dependency) 파일의 이름.
* **.PHONY**: 타겟 이름과 동일한 파일이 존재하더라도 타겟 레시피를 강제로 실행하도록 설정함.
* **VCS 실행 예시**:
  * `compile`: `vcs` 명령어를 통해 RTL 및 TB 파일을 컴파일하여 `simv` 생성함.
  * `sim`: 생성된 `simv`를 실행하며 `+UVM_TESTNAME`으로 실행할 테스트를 지정함.
  * `clean`: 컴파일 결과물 및 로그 파일 등을 삭제함.

class diagram으로 구조 어떻게 되어있는지 봤는데, 나중에 그릴 수도 있어야함.

## Class Diagram으로 봐야 하는 구조

나중에 직접 그릴 수 있어야 하는 구조는 아래 계층이다.

```text
uvm_test_top
└─ adder_test
   └─ env
      ├─ agt
      │  ├─ sqr
      │  ├─ drv
      │  └─ mon
      │     └─ ap : uvm_analysis_port #(adder_seq_item)
      ├─ scb
      │  └─ ap_imp
      ├─ cmp1
      │  └─ ap_imp_comp1
      ├─ cmp2
      │  └─ ap_imp_comp2
      └─ subs
         └─ analysis_export

tb_adder module
├─ adder_if
└─ adder DUT
```

| 계층 | 역할 |
| --- | --- |
| `tb_adder module` | 실제 DUT, interface, `run_test()`가 있는 Verilog module 영역 |
| `adder_test` | 전체 test scenario와 env 생성 담당 |
| `adder_env` | agent, scoreboard, 추가 component, subscriber를 묶는 검증 환경 |
| `adder_agent` | sequencer, driver, monitor를 묶는 interface 단위 검증 블록 |
| `adder_monitor` | interface에서 `a`, `b`, `y`를 관찰하고 transaction으로 변환 |
| `adder_scoreboard` | expected/actual 비교 담당 |
| `adder_component1`, `adder_component2` | monitor transaction을 추가로 수신하는 예제 consumer |
| `adder_subscriber` | coverage 같은 후처리 기능에 자주 쓰는 subscriber 예제 |

Class diagram을 그릴 때는 단순히 상속 관계만 그리는 것이 아니라, component hierarchy와 TLM 연결 방향을 같이 표시해야 한다.
특히 `monitor`의 analysis port가 `scoreboard`, `component1`, `component2`, `subscriber`로 갈라지는 broadcast 구조가 핵심이다.

## Analysis Port 다중 연결

`adder_monitor`에는 아래처럼 analysis port가 있다.

```systemverilog
uvm_analysis_port #(adder_seq_item) ap;
```

`adder_env.connect_phase()`에서는 monitor의 `ap`를 여러 수신점에 연결한다.

```systemverilog
agt.mon.ap.connect(scb.ap_imp);
agt.mon.ap.connect(cmp1.ap_imp_comp1);
agt.mon.ap.connect(cmp2.ap_imp_comp2);
agt.mon.ap.connect(subs.analysis_export);
```

| 연결 | 의미 |
| --- | --- |
| `agt.mon.ap -> scb.ap_imp` | scoreboard가 transaction을 받아 결과 비교 |
| `agt.mon.ap -> cmp1.ap_imp_comp1` | 첫 번째 추가 component가 같은 transaction 수신 |
| `agt.mon.ap -> cmp2.ap_imp_comp2` | 두 번째 추가 component가 같은 transaction 수신 |
| `agt.mon.ap -> subs.analysis_export` | subscriber가 같은 transaction 수신 |

`uvm_analysis_port`는 한 곳에만 연결되는 단일 point-to-point 통신이 아니라, 여러 consumer에게 같은 transaction을 broadcast할 수 있다.
따라서 monitor는 한 번 `ap.write(item)`을 호출하지만, 연결된 여러 component의 `write()`가 각각 실행된다.

## `uvm_analysis_imp`와 `uvm_subscriber` 차이

Scoreboard나 일반 component는 보통 `uvm_analysis_imp`를 직접 만들어 `write()`를 구현한다.
Subscriber는 `uvm_subscriber #(T)`를 상속해서 analysis 수신 구조를 더 간단히 만든다.

| 구분 | 일반 component + analysis imp | `uvm_subscriber` |
| --- | --- | --- |
| 상속 | `uvm_component`, `uvm_scoreboard` 등 | `uvm_subscriber #(adder_seq_item)` |
| 수신점 | 직접 만든 `uvm_analysis_imp` | base class가 제공하는 `analysis_export` |
| 핵심 method | `write(adder_seq_item item)` | `write(adder_seq_item t)` |
| 주 용도 | scoreboard, logger, custom consumer | coverage collector, subscriber형 consumer |

Subscriber는 coverage 기능에 자주 연결된다.
즉 monitor가 관찰한 transaction을 subscriber로 보내고, subscriber 내부에서 covergroup sample이나 통계 수집을 수행하는 구조로 확장할 수 있다.

## `uvm_subscriber` 작성 시 주의점

화면의 VCS 로그에서 아래와 같은 compile warning/error가 보였다.

```text
'uvm_subscriber#($unit::adder_seq_item)::write' in the derived class does
not match argument 't' for class method in base class
Please check and correct the argument names.
```

`uvm_subscriber`의 base class는 `write(T t)` 형태로 선언되어 있으므로, override할 때 simulator가 argument 이름까지 맞추라고 요구할 수 있다.
따라서 subscriber에서는 아래처럼 쓰는 편이 안전하다.

```systemverilog
class adder_subscriber extends uvm_subscriber #(adder_seq_item);
    `uvm_component_utils(adder_subscriber)

    virtual function void write(adder_seq_item t);
        `uvm_info(get_type_name(),
                  $sformatf("adder_subscriber: %s", t.convert2string()),
                  UVM_MEDIUM)
    endfunction
endclass
```

| 실수 | 수정 |
| --- | --- |
| `write(adder_seq_item item)`으로 작성 | `write(adder_seq_item t)`처럼 base 선언과 맞춤 |
| subscriber에 별도 `uvm_analysis_imp`를 또 만들려고 함 | `uvm_subscriber`가 제공하는 `analysis_export`를 사용 |
| coverage용 consumer를 scoreboard와 같은 역할로 봄 | scoreboard는 pass/fail 비교, subscriber는 coverage/관찰 후처리 쪽으로 구분 |

## Topology 출력으로 구조 확인

UVM report에서 `uvm_top.print_topology()`를 실행하면 실제 생성된 component 계층이 출력된다.
화면에서는 아래 구조가 확인됐다.

```text
uvm_test_top
  env
    agt
      drv
      mon
        ap
      sqr
    cmp1
      ap_imp_comp1
    cmp2
      ap_imp_comp2
    scb
      ap_imp
    subs
      analysis_imp
```

| 확인할 항목 | 의미 |
| --- | --- |
| `env.agt.drv` | driver가 agent 아래 생성됨 |
| `env.agt.mon.ap` | monitor가 analysis port를 가짐 |
| `env.scb.ap_imp` | scoreboard가 analysis imp를 가짐 |
| `env.cmp1.ap_imp_comp1` | 추가 component1도 수신 imp를 가짐 |
| `env.cmp2.ap_imp_comp2` | 추가 component2도 수신 imp를 가짐 |
| `env.subs.analysis_imp` | subscriber도 analysis 수신점으로 topology에 표시됨 |

Topology는 class diagram과 실제 simulation 계층이 일치하는지 확인하는 evidence로 쓸 수 있다.
Class diagram을 그릴 때도 topology 출력 순서를 참고하면 누락되는 component를 줄일 수 있다.

## Makefile 작성법

수업 후반에는 VCS 실행을 Makefile로 묶는 법을 확인했다.

```makefile
compile:
 vcs -full64 -sverilog -ntb_opts uvm-1.2 -timescale=1ns/1ps \
     -debug_access+all -kdb -lca -o simv ./rtl/adder.sv ./tb/tb_adder.sv

sim: compile
 ./simv +UVM_TESTNAME=adder_test +UVM_VERBOSITY=UVM_MEDIUM

clean:
 rm -rf simv* csrc *.log *.key *.h
```

| Makefile 요소 | 의미 |
| --- | --- |
| `target` | `compile`, `sim`, `clean`처럼 make로 실행할 이름 |
| `dependency` | target을 실행하기 전에 먼저 준비되어야 하는 target 또는 파일 |
| `recipe` | 실제 shell command |
| `TAB` | recipe 줄 앞에는 space가 아니라 반드시 tab이 필요 |

`sim: compile`은 `sim`을 실행하기 전에 `compile` target을 먼저 수행하라는 의미다.
따라서 `make sim`을 실행하면 compile 후 `./simv` 실행까지 이어진다.

## Makefile 사용 맥락 보강

수업 후반에는 Makefile을 왜 쓰는지와 `make <target>`이 내부적으로 어떤 순서로 동작하는지를 한 번 더 정리했다.

| 주제 | 정리 |
| --- | --- |
| 사용 목적 | VCS, Cadence Xcelium 같은 시뮬레이터/컴파일러 명령 반복 실행 자동화 |
| IDE와 차이 | 버튼 실행 대신 Linux/실무 환경에서 재현 가능한 command flow 관리 |
| target 탐색 | `make <target>` 실행 시 현재 위치의 `Makefile`에서 target 이름 검색 |
| dependency 처리 | target 실행 전 필요한 dependency를 먼저 처리 |
| check target 분리 | 필요한 검증 명령만 골라 실행할 수 있도록 target을 나누는 방식 |

정리하면 Makefile은 단순히 명령어를 짧게 치기 위한 파일이 아니라, target, dependency, recipe 관계로 컴파일과 시뮬레이션 절차를 재현 가능하게 관리하는 도구다.
UVM/VCS 실습에서는 `compile`, `sim`, `clean` target을 분리해 두면 컴파일 명령과 실행 명령을 매번 직접 입력하지 않아도 되고, 같은 구조를 다른 testbench에도 쉽게 재사용할 수 있다.

## Makefile Recipe와 실습 지시 보강

Makefile 작성 시 recipe 줄의 TAB과 간단한 변수 사용도 다시 강조됐다.
핵심은 복잡한 문법보다 compile/run 명령을 실수 없이 재사용할 수 있는 구조를 만드는 것이다.

| 주제 | 정리 |
| --- | --- |
| TAB 사용 | target 아래 recipe 줄은 반드시 TAB으로 시작 |
| Space 주의 | TAB 대신 space를 넣으면 recipe 실행 오류 발생 가능 |
| 변수 활용 | VCS compile/run 명령을 변수로 분리해 반복 수정 최소화 |
| 실습 방향 | 수업 내용을 바탕으로 RTL과 testbench를 직접 작성 |

이날 소스 정리는 작성 맥락을 구분해서 봐야 한다.
`260609_clk_adder_uvm`은 수업에서 교수님과 함께 확인한 clocked adder UVM 구조를 기준으로 정리한 것이고, `260609_ram`은 그 adder 예제를 바탕으로 같은 UVM 흐름을 RAM에 직접 적용해 본 개인 실습 소스다.
따라서 RAM 코드는 강의 중 교수님과 공동으로 작성한 원본이라기보다, 수업에서 다룬 sequence, driver, monitor, scoreboard, subscriber 연결 방식을 스스로 재구성해 본 결과로 보는 편이 맞다.

## `260609_clk_adder_uvm` Makefile 연결

현재 소스 폴더의 Makefile은 VCS 옵션, 출력 파일명, 소스 파일 목록, simulation 옵션을 변수로 분리했다.

| 변수 | 현재 값 | 의미 |
| --- | --- | --- |
| `VCS_OPTS` | `-full64 -sverilog -ntb_opts uvm-1.2 -timescale=1ns/10ps -debug_access+all -kdb -lca` | 64-bit SystemVerilog/UVM 1.2 compile, time unit/precision, debug DB 생성 옵션 |
| `VCS_OUTPUT` | `-o simv` | compile 결과 실행 파일 이름을 `simv`로 지정 |
| `VCS_SRC_FILES` | `./rtl/adder.sv ./tb/tb_adder.sv` | DUT와 UVM testbench compile 대상 |
| `SIMV_OPTS` | `+UVM_TESTNAME=adder_test +UVM_VERBOSITY=UVM_DEBUG +ntb_random_seed=1234` | 실행할 UVM test, debug verbosity, 고정 random seed |

| target | 동작 | 확인 포인트 |
| --- | --- | --- |
| `compile` | `vcs $(VCS_OPTS) $(VCS_SRC_FILES) $(VCS_OUTPUT)` 실행 | VCS compile만 수행 |
| `simv` | `compile`에 의존한 뒤 `./simv $(SIMV_OPTS)` 실행 | target 이름이 `sim`이 아니라 `simv` |
| `clean` | `simv*`, `csrc`, log/key/header 파일 삭제 | simulation 산출물 정리 |

Makefile에서 `compile`, `simv`, `clean` 아래 recipe 줄은 반드시 TAB으로 시작해야 한다.
space 8칸으로 시작하면 `make -n simv`에서 `missing separator (did you mean TAB instead of 8 spaces?)`가 발생할 수 있으므로, 현재 소스의 recipe 줄은 TAB으로 맞춰 두었다.

## `tb_adder.sv` 0608 대비 변경점

0608의 핵심 구조는 `monitor.ap -> scoreboard` 단일 수신이었다.
0609의 `tb_adder.sv`는 같은 DUT와 interface를 유지하면서 analysis 수신자를 늘려 broadcast 구조를 확인하는 쪽으로 바뀌었다.

| 변경 위치 | 0609에서 추가/변경된 내용 | 의미 |
| --- | --- | --- |
| `adder_component1` | `uvm_analysis_imp #(adder_seq_item, adder_component1) ap_imp_comp1`와 `write()` 추가 | scoreboard 외의 첫 번째 일반 consumer |
| `adder_component2` | `uvm_analysis_imp #(adder_seq_item, adder_component2) ap_imp_comp2`와 `write()` 추가 | 같은 transaction을 받는 두 번째 일반 consumer |
| `adder_subscriber` | `uvm_subscriber #(adder_seq_item)` 상속 class 추가 | coverage/후처리용 subscriber 예제 |
| `adder_scoreboard.write()` | log label을 `scoreboard:`로 바꾸고, 즉시 `Matched!`/`Mismatched!` 출력은 주석 처리 | 매 transaction마다 자세한 log를 찍기보다 pass/fail count와 report summary 중심으로 정리 |
| `adder_env` 멤버 | `cmp1`, `cmp2`, `subs` handle 추가 | env 안에 추가 consumer를 소유 |
| `adder_env.build_phase()` | `comp1`, `comp2`, `subs` instance 생성 | handle 이름은 `cmp1/cmp2`, component instance 이름은 `comp1/comp2` |
| `adder_env.connect_phase()` | `agt.mon.ap`를 `scb`, `cmp1`, `cmp2`, `subs.analysis_export`에 연결 | monitor가 만든 하나의 transaction stream을 네 수신자에게 broadcast |
| `rtl/adder.sv` | 0608과 차이 없음 | 변경 범위는 testbench/Makefile 중심 |

주의할 점은 `uvm_subscriber.write()` override에서 simulator가 base class의 `write(T t)` 인자명까지 요구할 수 있다는 점이다.
따라서 현재 `adder_subscriber.write()`는 `write(adder_seq_item t)` 형태로 맞춰 두었다.

## 핵심 정리

0609의 핵심은 UVM monitor가 만든 transaction을 analysis port로 여러 consumer에게 나눠 보내는 구조다.
Scoreboard는 pass/fail 비교를 담당하고, 추가 component나 subscriber는 같은 transaction을 받아 log, coverage, 통계 같은 후처리를 수행할 수 있다.

Class diagram을 그릴 때는 `test -> env -> agent -> monitor` 계층과 `monitor.ap -> scb/cmp1/cmp2/subs` TLM 연결을 함께 그려야 한다.
Makefile은 VCS `compile`, `simv`, `clean` 명령을 target 단위로 묶어서 반복 실행을 줄이는 용도이며, recipe 줄 앞의 TAB과 dependency 관계가 중요하다.
